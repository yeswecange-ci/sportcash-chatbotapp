1 — Architecture générale du système Vue d'ensemble des composants et du flux de données

Le système de tracking fonctionne comme une couche d'observation passive : votre Twilio Studio Flow continue de piloter la conversation normalement, et appelle l'API à chaque étape importante pour mémoriser ce qui se passe.

👤 Utilisateur WhatsApp
↓ Message entrant
📡 Twilio Studio – Flow principal
↓ HTTP Request Widget
🔴 POST /bot-tracking/twilio/incoming ← TOUJOURS en 1er
↓ Retourne conversation_id, is_client…
📡 Flow principal continue (menus, conditions…)
↓ À chaque choix de menu
🔴 POST /bot-tracking/twilio/menu-choice
↓ Transfert vers un Subflow
📡 Subflow (ex: Assistance Dépannage)
↓ Collecte d'infos (VIN, email…)
🔴 POST /bot-tracking/twilio/free-input
↓ Fin du Subflow / fin de conversation
🔴 POST /bot-tracking/twilio/complete

URL de base

Base URL (production Coolify) Remplacez https://votre-domaine.com par votre domaine configuré dans Coolify.
Tous les endpoints sont en HTTP POST sauf indication contraire.
Aucune authentification requise (les endpoints webhook sont publics mais protégés par Twilio).
# Base URL
https://votre-domaine.com

# 4 endpoints principaux
POST /bot-tracking/twilio/incoming
POST /bot-tracking/twilio/menu-choice
POST /bot-tracking/twilio/free-input
POST /bot-tracking/twilio/complete

# Endpoint générique (optionnel)
POST /bot-tracking/webhooks/event

Principe de fonctionnement

Quand ? Quel endpoint ? Ce que ça fait
Arrivée d'un message WhatsApp /twilio/incoming Crée ou reprend la conversation, retourne le contexte client
L'utilisateur choisit un menu /twilio/menu-choice Enregistre le choix dans les stats et l'historique
L'utilisateur tape du texte libre /twilio/free-input Stocke le texte, met à jour les données client
Fin de conversation / subflow terminé /twilio/complete Clôture la conversation, calcule la durée

2 — Référence des endpoints Paramètres, exemples et réponses pour chaque endpoint

2.1 — Réception message entrant

C'est le point d'entrée obligatoire. À appeler dès qu'un message WhatsApp arrive, avant tout autre traitement.

POST /bot-tracking/twilio/incoming

Crée une nouvelle conversation ou reprend une conversation active (moins de 24h). Retourne le contexte complet du client pour conditionner le flow.

Paramètres

ParamètreTypeObligatoireDescription
FromstringOUINuméro expéditeur Twilio. Format : whatsapp:+2250700000000
BodystringnonTexte du message reçu
MessageSidstringOUIIdentifiant unique du message Twilio (anti-doublon)
ProfileNamestringnonNom WhatsApp de l'utilisateur
NumMediaintegernonNombre de médias joints (images, audio…)
💡 Dans Twilio Studio Ces paramètres sont disponibles automatiquement via les variables Twilio : {{trigger.message.From}}, {{trigger.message.Body}}, {{trigger.message.MessageSid}}, etc.

Exemple de requête (HTTP Widget)

// Configuration du widget "Make HTTP Request" dans Studio
URL : https://votre-domaine.com/bot-tracking/twilio/incoming
Method : POST
Content-Type : application/x-www-form-urlencoded

// Body Parameters
From = {{trigger.message.From}}
Body = {{trigger.message.Body}}
MessageSid = {{trigger.message.MessageSid}}
ProfileName = {{trigger.message.ProfileName}}
NumMedia = {{trigger.message.NumMedia}}

Réponse JSON

{
  "success": true,
  "conversation_id": 142,
  "session_id": "session_6789abc...",
  "is_client": "true", // "true" | "false" | null
  "client_full_name": "KONAN Yao Bernard",
  "current_menu": "main_menu",
  "client_has_name": "true", // pour condition Twilio
  "client_status_known": "false",
  "has_media": "false",
  "status": "active"
}
⚠️ Important : is_client est une chaîne, pas un booléen Twilio Studio compare les variables comme des chaînes de caractères. Utilisez "true" et "false" entre guillemets dans vos conditions de Split widget.

Utilisation des variables retournées dans le flow

// Accéder aux données retournées (nom du widget : "track_incoming")
{{widgets.track_incoming.parsed.conversation_id}}
{{widgets.track_incoming.parsed.is_client}}
{{widgets.track_incoming.parsed.client_full_name}}
{{widgets.track_incoming.parsed.client_has_name}}
{{widgets.track_incoming.parsed.client_status_known}}

2.2 — Enregistrement choix de menu

À appeler chaque fois que l'utilisateur fait un choix dans un menu (principal ou sous-menu). S'adapte à n'importe quel nom de menu dans votre flow.

POST /bot-tracking/twilio/menu-choice

Enregistre le choix de menu, met à jour le chemin de navigation et alimente les statistiques dynamiques.

Paramètres

ParamètreTypeObligatoireDescription
conversation_idintegerOUIID retourné par /incoming
menu_choicestringOUIIdentifiant technique du menu (ex: menu_assistance, subflow_reclamation). Libre, ce sera le nom dans les stats.
user_inputstringnonValeur brute choisie par l'utilisateur (ex: "1", "2")
choice_labelstringnonLabel lisible du choix (ex: "Assistance dépannage"). Affiché dans les stats du dashboard.

Exemple de requête

URL : https://votre-domaine.com/bot-tracking/twilio/menu-choice
Method : POST

conversation_id = {{widgets.track_incoming.parsed.conversation_id}}
menu_choice = menu_principal
user_input = {{widgets.menu_principal.inbound.Body}}
choice_label = Assistance dépannage

Réponse JSON

{
  "success": true,
  "current_menu": "menu_principal",
  "menu_path": ["menu_principal"]
}

2.3 — Saisie libre / collecte de données

POST /bot-tracking/twilio/free-input

Enregistre toute saisie libre de l'utilisateur. Certains noms de widget déclenchent automatiquement la mise à jour des données client (nom, email, VIN, carte VIP).

Paramètres

ParamètreTypeObligatoireDescription
conversation_idintegerOUIID retourné par /incoming
user_inputstringnonTexte saisi par l'utilisateur
widget_namestringnonNom du widget Studio. Certains noms ont une signification spéciale (voir tableau ci-dessous).

Noms de widget avec traitement automatique

widget_nameEffet automatique
collect_nameSauvegarde le nom complet du client (ne remplace pas un nom existant)
collect_emailSauvegarde l'email du client
collect_vinSauvegarde le numéro VIN du véhicule
collect_carte_vipSauvegarde le numéro de carte VIP
check_clientMarque le client comme client ("1", "oui", "yes") ou non-client
autre nomEnregistre la saisie dans l'historique sans autre effet

Exemple — Collecte du nom

URL : https://votre-domaine.com/bot-tracking/twilio/free-input
Method : POST

conversation_id = {{widgets.track_incoming.parsed.conversation_id}}
user_input = {{widgets.ask_name.inbound.Body}}
widget_name = collect_name

2.4 — Finaliser une conversation

POST /bot-tracking/twilio/complete

Marque la conversation comme terminée. Calcule la durée totale. À appeler en fin de flow principal ou après un transfert vers un agent humain.

Paramètres

ParamètreTypeObligatoireDescription
conversation_idintegerOUIID retourné par /incoming

Exemple

URL : https://votre-domaine.com/bot-tracking/twilio/complete
Method : POST

conversation_id = {{widgets.track_incoming.parsed.conversation_id}}

Réponse

{ "success": true, "completed": true, "duration_seconds": 183 }

2.5 — Événement générique (optionnel)

POST /bot-tracking/webhooks/event

Endpoint universel pour enregistrer n'importe quel type d'événement personnalisé (erreur, timeout, transfert agent, etc.).

ParamètreObligatoireDescription

3 — Intégration dans le Flow principal Placement des widgets HTTP et variables à propager

Structure recommandée du Flow principal

1

Trigger (entrée du flow) → Widget HTTP : track_incoming

Premier widget après le Trigger. Connecte le numéro Twilio à votre base de données.

Nom du widget : track_incoming
URL : https://votre-domaine.com/bot-tracking/twilio/incoming
Method : POST

From = {{trigger.message.From}}
Body = {{trigger.message.Body}}
MessageSid = {{trigger.message.MessageSid}}
ProfileName = {{trigger.message.ProfileName}}
NumMedia = {{trigger.message.NumMedia}}
✅ Après ce widget, vous avez accès à : {{widgets.track_incoming.parsed.conversation_id}} — à stocker dans une variable de flow et passer à TOUS vos subflows.
2

Set Variables — Stocker le conversation_id

Utilisez un widget Set Variables ou Split Based On pour stocker le conversation_id dans une variable globale réutilisable partout.

// Widget "Set Variables" — Variables à créer
conversation_id = {{widgets.track_incoming.parsed.conversation_id}}
is_client = {{widgets.track_incoming.parsed.is_client}}
client_name = {{widgets.track_incoming.parsed.client_full_name}}
client_has_name = {{widgets.track_incoming.parsed.client_has_name}}
3

Condition sur is_client (Split Widget)

Utilisez les variables retournées pour personnaliser le parcours.

// Split Based On :
Variable : {{widgets.track_incoming.parsed.client_status_known}}
Conditions :
  "true" → parcours client identifié
  "false" → demander si client ou non
4

Affichage du menu principal → Widget HTTP : track_menu

Après chaque envoi du menu principal et réception de la réponse de l'utilisateur.

Nom du widget : track_menu_principal
URL : https://votre-domaine.com/bot-tracking/twilio/menu-choice
Method : POST

conversation_id = {{flow.variables.conversation_id}}
menu_choice = menu_principal
user_input = {{widgets.menu_principal.inbound.Body}}
choice_label = /* valeur humaine ex: "Assistance dépannage" */
💡 Astuce : choice_label dynamique Si votre menu utilise des numéros, mappez-les dans une variable :
choice_label = {{flow.variables.menu_label}} que vous définissez dans un Set Variables avant l'appel.
5

Avant chaque transfert vers un Subflow — Passer le conversation_id

C'est le paramètre le plus important. Sans lui, le subflow ne peut pas tracker.

// Widget "Run Subflow" — Parameters
conversation_id = {{flow.variables.conversation_id}}
is_client = {{flow.variables.is_client}}
client_name = {{flow.variables.client_name}}
6

Fin du flow → Widget HTTP : track_complete

À placer avant le widget End Flow.

Nom du widget : track_complete
URL : https://votre-domaine.com/bot-tracking/twilio/complete
Method : POST

conversation_id = {{flow.variables.conversation_id}}

Schéma visuel du flow principal

[Trigger WhatsApp]
    → [HTTP: track_incoming] POST /twilio/incoming
    → [Set Variables: conversation_id, is_client, client_name]
    → [Split: client_status_known ?]
        → [true] Afficher menu personnalisé
        → [false] Demander si client
            → [HTTP: track_free_input] POST /twilio/free-input (widget_name=check_client)
    → [Envoyer menu principal]
    → [HTTP: track_menu_principal] POST /twilio/menu-choice
    → [Run Subflow: passer conversation_id]
    → [HTTP: track_complete] POST /twilio/complete
    → [End Flow]

4 — Intégration dans les Subflows Règles et patterns pour chaque type de subflow

🔑 Règle fondamentale pour les Subflows Le conversation_id DOIT être passé depuis le Flow principal comme paramètre d'entrée. Ne jamais appeler /incoming depuis un subflow — cela créerait une conversation en double.

Paramètre d'entrée obligatoire dans chaque Subflow

// Dans les paramètres d'entrée du Subflow (Input Variables)
conversation_id // Passé par le flow parent
is_client // Optionnel, pour personnaliser le parcours

Pattern type : Subflow de collecte d'informations

Exemple : Subflow "Assistance dépannage" qui collecte le VIN et l'email.

[Début Subflow "Assistance Dépannage"]
  → [HTTP: track_sub_menu] POST /twilio/menu-choice
      menu_choice = subflow_assistance
      choice_label = Assistance Dépannage
  → [Envoyer message: "Quel est votre VIN ?"]
  → [Attendre réponse]
  → [HTTP: track_vin] POST /twilio/free-input
      widget_name = collect_vin
      user_input = réponse utilisateur
  → [Envoyer message: "Quel est votre email ?"]
  → [Attendre réponse]
  → [HTTP: track_email] POST /twilio/free-input
      widget_name = collect_email
  → [Fin Subflow — retour au Flow parent]

Pattern type : Subflow de menu secondaire

Exemple : Subflow "Mon Compte" avec des sous-choix.

1

Annoncer l'entrée dans le sous-menu

Nom : track_submenu_entry
URL : /bot-tracking/twilio/menu-choice
conversation_id = {{trigger.params.conversation_id}}
menu_choice = submenu_mon_compte
choice_label = Mon Compte
2

Tracker chaque choix du sous-menu

Nom : track_submenu_choice
URL : /bot-tracking/twilio/menu-choice
conversation_id = {{trigger.params.conversation_id}}
menu_choice = submenu_mon_compte
user_input = {{widgets.submenu_mon_compte.inbound.Body}}
choice_label = Solde points // ou "Historique achats", etc.

Récapitulatif par type de subflow

Type de subflow Endpoint(s) à utiliser Paramètre clé
Menu secondaire /menu-choice menu_choice = nom du sous-menu
choice_label = label du choix
Collecte de données /free-input pour chaque collecte widget_name = collect_name, collect_vin, etc.
FAQ / Informations /menu-choice à l'entrée menu_choice = subflow_faq
Transfert vers agent /complete après le transfert Statut = completed automatique
Réclamation /menu-choice + /free-input Enregistrer le motif de réclamation en texte libre
⚠️ Ne pas appeler /complete depuis un Subflow intermédiaire Appelez /complete uniquement depuis le Flow principal, à la toute fin. Si vous l'appelez depuis un subflow intermédiaire, la durée sera calculée trop tôt et la conversation sera fermée prématurément.

Cas 1 : Nouvel utilisateur inconnu

Un numéro qui n'a jamais contacté le bot. Le système crée une conversation et retourne client_status_known = "false".

ProfileName = Konan Yao

// Appel à /free-input

// Appel à /menu-choice

// Appel à /free-input avec widget_name custom

// Sous-choix dans le subflow réclamation
// Appel à /menu-choice

7 — Bonnes pratiques & erreurs fréquentes Ce qu'il faut faire, ce qu'il faut éviter

✅ Ce qu'il faut faire

Toujours appeler /incoming en tout premier Avant toute logique de flow, même avant d'envoyer un message. C'est lui qui initialise la conversation.
Stocker conversation_id dans une variable de flow flow.variables.conversation_id est accessible dans tout le flow et dans tous les subflows. Ne pas le recalculer.
Donner un choice_label lisible à chaque menu_choice Le choice_label est ce qui apparaît dans les graphiques du dashboard. "Assistance dépannage" est bien meilleur que "1".
Utiliser des menu_choice cohérents entre vos flows Si le même menu existe dans plusieurs subflows, utilisez le même nom. Les stats seront regroupées automatiquement.

❌ Ce qu'il faut éviter

Ne JAMAIS appeler /incoming depuis un subflow Cela crée une deuxième conversation pour le même utilisateur, dupliquant les stats.
Ne JAMAIS appeler /complete au milieu d'un subflow La conversation se ferme et la durée est calculée trop tôt. Appelez /complete uniquement dans le flow principal, juste avant End Flow.
Ne pas oublier le conversation_id dans les subflows Si vous n'avez pas de conversation_id dans un subflow, les appels API retourneront 404 et rien ne sera tracké.
⚠️ Gérer les erreurs HTTP dans Twilio Studio Configurez la connexion "Failed" de chaque widget HTTP pour continuer le flow même si l'API est inaccessible. Le tracking ne doit jamais bloquer la conversation utilisateur.
